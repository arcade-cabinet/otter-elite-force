Entity Definition Architecture

Everything about an entity lives in one place. TypeScript modules as the single source of truth.

Core Principle

Each game entity (unit, building, resource, terrain tile, portrait) is a single TypeScript file that contains ALL of its data: visual appearance (ASCII sprite frames), gameplay stats, economic cost, AI behavior profile, Koota trait composition, drop tables, and unlock conditions. No fragmentation across directories.

Data Flow at Runtime

Boot → Play Pipeline
1. BootScene.create() │ ├─ Import ALL entity definitions (tree-shaken by Vite) ├─ Detect device scale factor (window.devicePixelRatio + canvas size) ├─ For each definition with sprite data: │ ├─ Create offscreen Canvas at (gridSize × scaleFactor) │ ├─ Draw pixels from ASCII grid + PALETTE │ ├─ this.textures.addCanvas(entity.id, canvas) │ └─ (For animated: add frame rects to atlas) └─ All textures registered → transition to MenuScene 2. GameScene.create(missionId) │ ├─ Import mission definition (terrain grid + placements) ├─ mapPainter.paintTerrain(terrainGrid, scaleFactor) │ ├─ Create large offscreen Canvas (mapW × mapH × tileSize × scale) │ ├─ For each tile: draw terrain sprite from definition │ ├─ Add noise/variation per terrain type │ └─ Register as 'terrain-bg' texture ├─ For each placement in mission.entities: │ ├─ Look up definition from ALL_UNITS / ALL_BUILDINGS / etc. │ └─ spawner.spawn(world, definition, x, y, faction) │ ├─ world.spawn(Position, Health, Attack, Armor, ...) │ ├─ Apply ALL traits from definition.traits │ └─ Sync layer auto-creates Phaser sprite └─ Initialize scenario engine with mission.triggers
